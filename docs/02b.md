## Message Type Reference

### Session Management

| Type | Message | Direction | Response |
|------|---------|-----------|----------|
| VDSM_REQUEST_HELLO | vdsm_RequestHello | vdSM → vDC | vdc_ResponseHello |
| VDC_RESPONSE_HELLO | vdc_ResponseHello | vDC → vdSM | N/A |
| VDSM_SEND_BYE | vdsm_SendBye | vdSM → vDC | None |

### vdsm_RequestHello
Message metadata
| Property | Value |
|---|---|
| Name | vdsm_RequestHello |
| Topic | session/hello/request |
| Direction | vdSM → vDC |
| Version | 1.0 |
| Summary | Open a session or handshake with the vDC. Contains client identification and protocol details. |

Fields
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| requestId | string | yes | Client-generated unique request identifier. Used to correlate responses. | "req-1234" |
| clientId | string | yes | Persistent id for the vdSM instance (can be UUID). | "vdsm-01" |
| protocolVersion | string | yes | Client protocol version (major.minor). | "2.0" |
| clientCapabilities | object | no | Key/value map of client capabilities (optional). | {"scenes":true} |
| timestamp | string (ISO-8601) | no | Client local time for diagnostics. | "2026-01-09T12:34:56Z" |

JSON example
```json
{
  "requestId": "req-1234",
  "clientId": "vdsm-01",
  "protocolVersion": "2.0",
  "clientCapabilities": {"scenes": true},
  "timestamp": "2026-01-09T12:34:56Z"
}
```

Notes
- Server responds with vdc_ResponseHello (or GenericResponse with an error).
- requestId should be unique per outstanding request; the response echoes it.
- If protocolVersion is incompatible the server will reply with an error and recommended version.

### vdc_ResponseHello
Message metadata
| Property | Value |
|---|---|
| Name | vdc_ResponseHello |
| Topic | session/hello/response |
| Direction | vDC → vdSM |
| Version | 1.0 |
| Summary | Response to RequestHello containing server identification and negotiated parameters. |

Fields
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| requestId | string | yes | Echoes the requestId from the RequestHello. | "req-1234" |
| serverId | string | yes | Server identifier. | "vdc-main-01" |
| protocolVersion | string | yes | Negotiated protocol version. | "2.0" |
| serverTime | string (ISO-8601) | yes | Server timestamp when response generated. | "2026-01-09T12:34:57Z" |
| status | string | yes | "ok" or error code. | "ok" |
| details | object | no | Additional details or server capabilities. | {"maxPayload":1048576} |

JSON example
```json
{
  "requestId": "req-1234",
  "serverId": "vdc-main-01",
  "protocolVersion": "2.0",
  "serverTime": "2026-01-09T12:34:57Z",
  "status": "ok",
  "details": {"maxPayload": 1048576}
}
```

Notes
- If status != "ok" the details field should include explanation and recommended action.
- After a successful hello the session is considered established for the requesting clientId.

### vdsm_RequestGetProperty
Message metadata
| Property | Value |
|---|---|
| Name | vdsm_RequestGetProperty |
| Topic | property/get/request |
| Direction | vdSM → vDC |
| Version | 1.0 |
| Summary | Request one or more property values from the vDC. |

Fields
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| requestId | string | yes | Unique request id. | "get-001" |
| propertyPath | string | yes | Dot-separated path to the property (or wildcard). | "devices.device123.channel.1.value" |
| attributes | array of string | no | Optional attributes to include (e.g., ["timestamp","quality"]). | ["timestamp"] |

JSON example
```json
{
  "requestId": "get-001",
  "propertyPath": "devices.device123.channel.1.value",
  "attributes": ["timestamp"]
}
```

Notes
- Server responds with vdc_ResponseGetProperty or with an error (GenericResponse).
- propertyPath supports exact paths and may support wildcards depending on server.

### vdc_ResponseGetProperty
Message metadata
| Property | Value |
|---|---|
| Name | vdc_ResponseGetProperty |
| Topic | property/get/response |
| Direction | vDC → vdSM |
| Version | 1.0 |
| Summary | Response containing property value(s) and optional metadata. |

Fields
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| requestId | string | yes | Echo of requestId. | "get-001" |
| results | array | yes | Array of property result objects. | see example |
| status | string | yes | "ok" or error code. | "ok" |
| timestamp | string (ISO-8601) | yes | Server time. | "2026-01-09T12:35:00Z" |

Within results objects:
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| path | string | yes | Property path returned. | "devices.device123.channel.1.value" |
| value | any | yes | Property value. | 75 |
| type | string | no | Data type of the value. | "integer" |
| attributes | object | no | Additional metadata (timestamp, quality). | {"timestamp":"2026-01-09T12:34:59Z"} |

JSON example
```json
{
  "requestId": "get-001",
  "status": "ok",
  "timestamp": "2026-01-09T12:35:00Z",
  "results": [
    {
      "path": "devices.device123.channel.1.value",
      "value": 75,
      "type": "integer",
      "attributes": {"timestamp": "2026-01-09T12:34:59Z"}
    }
  ]
}
```

Notes
- If some requested properties are not found, results may include entries with an error field describing the issue.
- For bulk queries the results array may contain many entries.

### vdsm_RequestSetProperty
Message metadata
| Property | Value |
|---|---|

Fields
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| requestId | string | yes | Unique request id. | "set-001" |
| changes | array | yes | Array of change objects {path, value, type?, priority?}. | see example |

| Field | Type | Required | Description | Example |

JSON example
```json

  "transactional": true,
  "changes": [

      "path": "devices.device123.channel.1.value",
      "value": 0.5,
      "type": "float",
      "priority": 100
    }
  ]
}
```

Notes
- Server responds with GenericResponse indicating success/failure per change (or an aggregated status).
- If transactional=true and one change fails, server should rollback if possible.

### vdc_SendPushProperty
Message metadata
| Property | Value |
|---|---|
| Name | vdc_SendPushProperty |
| Topic | property/push |
| Direction | vDC → vdSM |
| Version | 1.0 |
| Summary | Server push to client to notify property changes (unsolicited). |

Fields
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| timestamp | string (ISO-8601) | yes | Time of change. | "2026-01-09T12:35:05Z" |
| updates | array | yes | Array of updates: {path, value, type, attributes}. | see example |

Update object (as in ResponseGetProperty results)

JSON example
```json
{
  "timestamp": "2026-01-09T12:35:05Z",
  "updates": [
    {
      "path": "devices.device123.channel.1.value",
      "value": 76,
      "type": "integer",
      "attributes": {"timestamp": "2026-01-09T12:35:05Z"}
    }
  ]
}
```

Notes
- Push messages are typically not acknowledged; server expects clients to accept or request correction/errors separately.
- If an error occurs delivering a push the server may emit an Error message or retry depending on transport.

### vdsm_RequestGenericRequest
Message metadata
| Property | Value |
|---|---|
| Name | vdsm_RequestGenericRequest |
| Topic | generic/request |
| Direction | vdSM → vDC |
| Version | 2.0 |
| Summary | Generic RPC-style request to invoke a named operation or API extension on the vDC. Server replies with GenericResponse. |

Fields
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| requestId | string | yes | Unique request identifier. | "gr-001" |
| method | string | yes | Method name to invoke (extension point). | "device.reboot" |
| params | object | no | Method parameters. | {"deviceId":"device123","force":true} |
| timeout | integer (ms) | no | Optional client timeout. | 5000 |

JSON example
```json
{
  "requestId": "gr-001",
  "method": "device.reboot",
  "params": {"deviceId": "device123", "force": true},
  "timeout": 5000
}
```

Notes
- The server replies with GenericResponse which includes status, optional result object or error.
- This mechanism is used to extend the protocol without changing message definitions.

## Common response: GenericResponse

Message metadata
| Property | Value |
|---|---|
| Name | GenericResponse |
| Topic | response/generic |
| Direction | vDC → vdSM |
| Version | 1.0 |
| Summary | Standard response envelope for requests which do not have a dedicated response message. |

Fields
| Field | Type | Required | Description | Example |
|---|---:|:---:|---|---|
| requestId | string | yes | The request id being answered. | "set-001" |
| status | string | yes | "ok" or error short code. | "ok" |
| code | integer | no | Numeric status code (optional). | 200 |
| result | object | no | Result payload for success. | {"applied":1} |
| error | object | no | Error details if status != "ok". | {"message":"not found","details":{}} |
| timestamp | string | yes | Server response time. | "2026-01-09T12:53:00Z" |

JSON example (success)
```json
{
  "requestId": "set-001",
  "status": "ok",
  "code": 200,
  "result": {"applied": 1},
  "timestamp": "2026-01-09T12:53:00Z"
}
```

JSON example (error)
```json
{
  "requestId": "set-001",
  "status": "error",
  "code": 404,
  "error": {"message": "device not found"},
  "timestamp": "2026-01-09T12:53:00Z"
}
```

Notes
- GenericResponse is used for RequestSetProperty, GenericRequest and other operations that use a common response envelope.
- For batch operations the result may contain per-item statuses.
